Copy verification modes
How INGESTO checks your footage after copying each card.
FAST
Copy only — no verification. Files are written to the destination and nothing is checked afterwards. Fastest, but a silent transfer error or a truncated file would go unnoticed. Use only for throwaway or non-critical material, or a second redundant copy you'll verify elsewhere.
VERIFIED
Copy + size check. After copying, INGESTO compares the byte size of each source and destination file. Catches incomplete or truncated copies (disconnected drive, full disk, interrupted write) at almost no time cost. It does not detect silent bit-level corruption, because the content itself isn't read back.
SECURE
Copy + xxHash content check. INGESTO computes a 64-bit xxHash fingerprint of the source and of the written copy and compares them. Detects any data corruption down to a single flipped bit. This is the broadcast/archive-grade mode (same family of checks as Silverstack / YoYotta). Slower — it reads every file back — and noticeably so on cards with thousands of tiny files.
PRO
Copy + checksum content check + checksum list. Does everything SECURE does, using the algorithm you pick (xxHash64 / xxHash128 / MD5), and additionally writes a checksum list in each destination folder (.xxh / .xxh3 / .md5, TeraCopy-style) so the copy can be re-verified later — with INGESTO, TeraCopy, xxhsum -c / md5sum -c, or your own tool. For advanced archival/hand-off workflows. SECURE already guarantees copy integrity for almost everyone; MD5 is notably slower.
PRO · Double read
Optional in PRO — detects a failing card. After verifying the copies, INGESTO reads the source a second time and checks it still returns the same data. A card that gives different bytes on re-read is dying, and those files are flagged as suspect (and left out of the checksum list). Limit: the second read only reaches the card if the data no longer sits in the OS RAM cache — which is the case on real offloads (footage larger than your RAM), but not guaranteed on a small test copy on a high-RAM machine. It roughly doubles verify time. Hard read errors are always caught, cache or not.
Counter
How the ingest number for each card is chosen.
Auto (default). INGESTO scans your destinations and picks the next free number, so cards get consecutive numbers even across sessions. It rescans automatically whenever a destination is added or a copy finishes.
Manual override. Typing a number, or using +/−, locks the counter to that value. It stays locked until you click Auto again, which rescans the destinations and returns to automatic mode.
Auto-detect unformatted card
Knowing whether a card has already been ingested.
When ON, INGESTO writes a small log file (.ingesto.json) on the source card right after copying. It records which files were copied, so the next time you insert the card INGESTO can tell you it has already been ingested — handy to avoid a double copy or to catch a card that wasn't reformatted. This requires write access to the card.
When OFF (default), INGESTO never writes anything on your source cards — not even a temporary check file. The card is only ever read.
Double source read
Detect a card that's starting to fail.
When ON (PRO only), after checking the copies INGESTO reads the source a second time and confirms it still returns the same data. A card that gives different bytes on re-read is dying — those files are flagged as suspect and left out of the checksum list. Hard read errors are always caught.
Limit to know: the second read only reaches the card if its data no longer sits in the OS RAM cache. That's the case on real offloads (footage larger than your RAM), but not guaranteed on a small test copy on a high-RAM machine. It roughly doubles verify time.
Ingest Report
A running log of every card, written to the destination.
When ON, INGESTO writes a self-contained INGESTO_report.html at the root of each destination and adds a line after every ingest: #, date, card, operator, camera, mode, files, size, result, note. Open it any time with View; it opens in your browser and has an Export PDF button (clean printable version).
The file lives in the destination and carries its own data, so if you quit mid-project and relaunch on the same destination, the log is picked up and continued. It also captures ingests made in Shooter mode. It's a log of events — re-ingesting a card adds a new line.
Formats written to each destination
HTML — to read and print (INGESTO_report.html)
CSV — for spreadsheets / databases
JSON — for scripts and other tools
Checksum
Fingerprints written to each destination for later re-verification.
Available in SECURE and PRO. INGESTO writes the per-file fingerprints it already computed during the ingest, so another tool can re-verify the copies later, on another machine or months down the line.
Formats written
Checksum list — TeraCopy-style .xxh/.md5, re-checkable in TeraCopy (SECURE & PRO)
MHL — classic .mhl manifest read by Silverstack, YoYotta, OffShoot (PRO only)
About MHL: written only when the PRO checksum algorithm is MD5 or xxHash64. With xxHash128 (the PRO default) no MHL is produced. For a first test prefer MD5: the xxHash64 byte order in MHL can't be guaranteed against every tool and should be validated on a real ingest in Silverstack or YoYotta.
PRO mode — advanced use
Not recommended for most users.
PRO does everything SECURE does, and also writes a checksum list in each destination folder for later re-verification. It's meant for specific archival / hand-off workflows where you carry those lists into another tool.

For normal ingest, SECURE already guarantees your copies bit-for-bit — you don't need PRO. PRO is slower (especially in MD5), adds extra files to your folders, and assumes you know why you want checksum lists.

Settings
Phone notification (ntfy)
Send a push to your phone when an ingest finishes
Install the ntfy app on your phone, subscribe to this exact topic, and you'll be notified when an ingest finishes — even away from the computer. Pick a long, hard-to-guess topic name (it acts as a password on the public server).
Post-ingest command
Runs a shell command after each ingest (not when canceled). Leave empty to disable. Variables: {destPath} · {card} · {count} · {bytes} · {result} (quote them in your command, e.g. "{destPath}"). ⚠ This executes exactly what you type — only enter commands you trust.
